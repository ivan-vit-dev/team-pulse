Architecture Overview

Guiding constraints and decisions — read this before touching the code.

Auth model — server-first

Client SDK sign-in → ID token POSTed to /api/auth/session (Node runtime) → firebase-admin verifies, ensures the Firestore profile exists, mints an httpOnly session cookie. getCurrentUser() in src/lib/auth/session.ts is the authoritative check; useAuth/AuthProvider client state is UX only.

Firestore access

All mutations go through Server Actions calling repository modules in src/lib/* (Admin SDK). Client code never writes Firestore directly — the only direct client Firebase use is Storage uploads (avatars, logos, action photos). Authorization is TypeScript-first via requireTeamAdmin().

Youth privacy — document boundary

Firestore has no field-level read security, so players split into a world-readable public doc and players/{id}/private/profile (real name, birthdate — admin-only). isYouth is recomputed server-side on every write; youth avatars force-nulled.

Notifications pipeline

Five Firestore triggers in functions/src/index.ts funnel through a shared notifyUsers(): per-category preference gate (skips the in-app write entirely), in-app inbox doc, FCM multicast with invalid-token pruning.

Rules are defense-in-depth

firestore.rules/storage.rules encode the intended access model (and keep a future real-time-listener feature from being wide open), but real writes go through the Admin SDK which bypasses them. Both layers must stay in sync.

Schema evolution

Pre-launch convention: full-replace renames (likes → reactions, notes → description) plus defensive reads that default missing fields (kind, isPinned, isArchived, startDate). No migration scripts.

Youth Privacy Engine

src/lib/players/ — load-bearing, not a style choice.

youth.ts + player-repository.ts

YOUTH_AGE_THRESHOLD = 18 (a constant today; placeholder for Remote Config). isYouth is computed from birthdate server-side and re-checked on every write; whenever it's true the repository forces avatarURL to null regardless of client input. Public data (PlayerPublic: pseudonym, jersey number, isYouth flag) lives in players/{id}; real name and birthdate live in players/{id}/private/profile, readable only by that team's admins. Any new surface that shows a player must consume PlayerPublic — never re-derive a "safe" view ad hoc.

Security Rules

Defense-in-depth: Server Actions with the Admin SDK bypass these, but the rules encode the intended model. Keep both layers in sync.

Firestore (firestore.rules)

players/{playerId} + private subdoc Public doc world-readable; private/profile readable only via isTeamAdmin(). The document split IS the privacy mechanism.
teams / seasons / actions / media Read: public. Create/update/delete: isTeamAdmin(teamId) (flat check — real writes never go through rules anyway).
comments Create: signed-in author only. Update (pinning): team admin. Delete: author or admin.
users/{uid}/notifications Read/mark-read: owner only. allow create, delete: if false — Cloud Functions (Admin SDK) are the only writers.
reports Create: signed-in reporter with status == "pending". Read/resolve: the reported team's admins (or the reporter reading their own).

Cloud Functions

functions/src/index.ts — separate package, Firestore triggers v2. All five funnel through the shared notifyUsers() helper.

FunctionTriggerPurpose
onActionCreatedonCreate actions/{id}Notifies every follower of the team (excludes the creator).
onActionUpdatedonUpdate actions/{id}Diffs MEANINGFUL_ACTION_FIELDS (title, type, opponent, date, time, location, competition, isHome, result, description) so reactions/squad tweaks don't spam; excludes the editor via updatedBy.
onAdminInviteCreatedonCreate teamAdminInvites/{id}Email → getUserByEmail; silently no-ops if that email has no account yet.
onFollowInviteCreatedonCreate teamFollowInvites/{id}Same email matching for follow invites.
onCommentCreatedonCreate comments/{id}Only fires for genuine replies (parentCommentId set); skips self-replies.
notifyUsers() — two independent gates

1. Per-category gate (notificationPreferences.categories): a disabled category skips the in-app inbox write entirely, not just push.
2. Channel gate (notificationPreferences.push): controls FCM multicast only. Invalid/expired tokens are pruned from the user doc after each send. The email channel switch exists on the shape but no email is sent yet (backlog).

Patterns & Gotchas

Base UI composition — render prop, never asChild

shadcn here is built on Base UI, not Radix
// WRONG — Button's children are discarded when render is set
<Button render={<Link href="/teams/new" />}>Create</Button>

// CORRECT — the label lives on the element inside render
<Button render={<Link href="/teams/new">Create</Link>} />

Never export type from a "use server" file

Breaks every Server Action app-wide, invisible to tsc/eslint/build

Files with "use server" may only export async functions. A type re-export poisons the module at runtime even though all static checks pass. Keep shared types in src/lib/types/ and import them where needed.

This is NOT the Next.js you know

Next.js 16 — check node_modules/next/dist/docs before assuming an API

Breaking changes vs. training-data-era Next.js. Example already hit: middleware.ts was replaced by proxy.ts. See AGENTS.md.

Timestamps never cross the client boundary

Strip with omit() before passing to Client Components

Firestore Timestamp instances can't serialize into Client Component props. Every page maps entities through omit(entity, "createdAt", "updatedAt") and components declare ClientSafeX = Omit<X, "createdAt" | "updatedAt">.

No router.refresh() after router.push()

Confirmed via live testing

Calling refresh() immediately after push() interrupts the in-flight transition. A push to a route not in the router cache already fetches a fresh RSC payload. (Standalone router.refresh() after an in-place mutation is the normal pattern.)

Opacity with CSS-variable colors

bg-primary/50 breaks with oklch tokens

Use color-mix(in oklch, var(--token) 50%, transparent). See APP_DESIGN.md.

Defensive read, no migration scripts

Pre-launch schema convention

Renames are full replaces (likedByUids → reactions, notes → description); newer optional fields get defaults at the repository read layer (kind: "image", isPinned: false, isArchived: false). Read via action.reactions ?? {}-style fallbacks in older call sites.

Youth privacy in every new feature

Reuse PlayerPublic — never re-derive a "safe" view

Any comment/media/timeline work that surfaces a player must consume PlayerPublic. Youth players: pseudonym only, no photo, no personal info in public views, YouthPrivacyBadge where identity could leak.
