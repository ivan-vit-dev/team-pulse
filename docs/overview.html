⚽ Youth & Amateur Football Platform

TeamPulse

A responsive web app connecting fans, parents, and supporters with amateur and youth football teams — season timelines, match results, comments, media, reactions, and push notifications, with youth-player privacy built into the data model itself.

Next.js 16 React 19 TypeScript 5 Firebase Tailwind v4 shadcn/ui (Base UI) next-intl · en + cs FCM Push Youth privacy first
About

What is this app?

TeamPulse lets team admins publish seasons, matches, trainings, and tournaments, while fans follow their teams and live the season through a timeline of results, photos, videos, comments, and reactions.

Target audience

Fans, parents, and supporters of amateur and youth football teams, plus the volunteer team admins who run those teams. English and Czech localization, mobile-first responsive UI.

Core concept

Each team publishes a season timeline of "actions" (matches, trainings, tournaments, cups). Fans follow teams, get a personalized home feed, and engage on each action with comments, media, and one of five football-flavored emoji reactions (⚽🔥👏❤️😂).

Youth privacy

Players under 18 appear publicly only as a pseudonym — no photo, no real name, no birthdate. This is enforced as a document boundary in Firestore (public vs. admin-only private doc), not just a UI filter, and re-checked server-side on every write.

Status

All 7 MVP phases are implemented, plus FR-level SRS follow-ups (video links, pinnable media, date-bounded seasons with archive lifecycle, explicit opponent/description fields). No monetization — a community product. Remaining backlog: email notification channel, user blocking.

User Roles

Who uses it

Deliberately flat: no platform-wide moderator or super-admin role. A team's own admins moderate that team's content.

Features

Main capabilities

Seven MVP phases, all shipped, plus post-MVP requirement follow-ups.

🛡️

Team Management

Create and brand teams (logo, colors, socials). Team-scoped theming: a team's colors skin its own pages, never the app chrome. Co-admin invites by email with accept/decline.

🔒

Players with Youth Privacy

Public doc (pseudonym, jersey number) split from a private doc (real name, birthdate). isYouth computed server-side from birthdate; avatars force-nulled for youth players regardless of what the client sends.

📅

Seasons & Lifecycle

Date-bounded seasons (optional start/end) with an archive state: archived seasons stay browsable but are closed for new content and can't be set active. Exactly one active season per team, enforced transactionally.

⚽

Actions

Matches, trainings, tournaments, cups. Explicit opponent and description fields, competition, home/away, kickoff time, result score, and a squad call-up list drawn from the roster.

📜

Season Timeline

The signature surface: next action highlighted with a floodlight pulse, upcoming actions muted, past results as torn ticket-stub cards, cursor-paginated history.

➕

Follow System

Follow/unfollow from the public team page, a personalized signed-in home feed, fan-facing follow invites by email, and a share-follow-link button for admins.

💬

Comments

Single-level reply threading (a reply to a reply still anchors to the top-level comment). Admins can pin comments; authors and admins can delete.

📸

Media

Admin-uploaded photos plus external YouTube/Vimeo video links, rendered as privacy-friendly embeds. Unknown hosts rejected at write time. Media can be pinned to the top of the gallery.

🔥

Reactions

Five football-flavored emoji (⚽🔥👏❤️😂), one reaction per user per action — picking a new one replaces the old, stored as a single dot-path field update.

🔔

Notifications

Five Cloud Function triggers (new action, meaningful action edits, admin invite, follow invite, comment reply) fan out FCM push and a persisted in-app inbox, gated by per-category preferences and a push channel switch.

🚩

Moderation Console

Fans report comments, media, teams, or players. The team's own admins review pending reports with a content preview and resolve, dismiss, or remove the content.

🌐

i18n & Theming

Full English + Czech catalogs via next-intl, locale-prefixed URLs, light and dark themes, and an ambient "matchday under lights" visual identity.

Architecture

Key decisions

Constraints and patterns that shape all development decisions.

  • Session-cookie auth — server-first, not client-only Client signs in with the Firebase client SDK, POSTs the ID token to /api/auth/session (Node runtime), which verifies it, ensures the Firestore profile exists, and mints an httpOnly session cookie via firebase-admin. Server Components read auth via getCurrentUser(); client auth state is UX-only.
  • All mutations via Server Actions using the Admin SDK Client code never writes to Firestore directly (the sole exception: Storage uploads for avatars/logos/photos). Authorization is enforced in TypeScript first (requireTeamAdmin()); Firestore rules encode the same model as defense-in-depth.
  • Youth privacy is a document boundary, not a field filter Firestore has no field-level read security, so each player splits into a public doc and players/{id}/private/profile (admin-only). isYouth is recomputed from birthdate on every write and youth avatars are forced to null server-side.
  • Profile creation inline in the session route, not an onCreate trigger Avoids a second deploy target and the eventually-consistent race where the client lands on /settings before a background trigger wrote the profile doc. ensureUserProfile() is idempotent.
  • Emulator-first local development The Local Emulator Suite is the default target — .env.local.example ships demo values that work with no real Firebase project. Both SDK init files branch on emulator env vars.
  • proxy.ts, not middleware.ts Next.js 16 replaced middleware with proxy. src/proxy.ts composes next-intl locale routing with a cheap auth-cookie presence check; the authoritative auth check happens in the protected layout.
  • Schema evolution: defensive read, no migration scripts Pre-launch, fields are renamed or replaced outright (e.g. likes → reactions, notes → description); reads default missing fields (kind, isPinned, isArchived) instead of running migrations.
